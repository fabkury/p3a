# p3a v1.0 Readiness Audit

**Date:** 2026-06-02 · **Audited version:** 0.10.2 · **Scope:** first-party firmware (`main/` + `components/`, excluding vendored `ugfx`/`managed_components`) and the Web UI (`webui/`).

**Lenses:** correctness & robustness · API/protocol stability · cleanup/refactor. Security was folded in opportunistically where it overlapped these.

**Method:** a multi-agent sweep — one reviewer per subsystem (16 units) reading the code directly, every finding then re-checked by an independent adversarial verifier, plus three cross-cutting passes (API/protocol stability, cross-component duplication, and a completeness critic that reconciled the code against the known-issue docs). **All severities below are the verifier-corrected values**, not the original reviewer's. One finding was rejected on verification and is omitted. Items the verifier downgraded are marked where it matters.

---

## Bottom line

p3a is genuinely mature, well-commented code: mutex discipline on the hot paths is mostly correct, allocations overwhelmingly use the PSRAM-first-with-internal-fallback pattern, error/cleanup paths are usually present, and the newer storage formats (pin_lists, the active-playset snapshot) are well-designed. **No unbounded leaks or use-after-frees were found in steady-state playback.** The 1.0 gate is therefore *not* a rewrite — it's a focused punch-list.

That said, **1.0 is not justified as-is.** The blockers are concrete:

1. A **documented, reproducible hard-crash-and-reboot** (SDIO RX OOM) is still unmitigated in the shipping config.
2. **✅ Fixed in `5cd491b7` (2026-06-02).** Two **memory-safety bugs reachable from untrusted input** — a heap overflow from a non-conforming Giphy API response, and a stack overflow in the captive-portal DNS server reachable by any client on the setup AP.
3. The **public contracts that 1.0 promises to keep stable** (REST URL surface, the API-version mechanism, several on-disk/NVS formats) are inconsistent or outright broken today, and freezing them in their current state would bake in the inconsistency.

> **Status (2026-06-02):** blocker #2 is resolved — B2/B3 plus two companion body-read fixes shipped in `5cd491b7`. **B1 (SDIO) is now the only open Tier-0 item**, and it needs a `sdkconfig` change + on-hardware soak rather than a code edit.

There is also **zero automated test or build CI** for an ~80k-line codebase about to commit to stability guarantees.

Recommended path to 1.0: clear **Tier 0**, make the **Tier 1 API/protocol freeze decisions**, stand up **build CI**, then ship. Tiers 2–3 can trail into 1.0.x.

| Tier | Meaning | Count* |
|------|---------|-------|
| **0 — Blocker** | Crash / memory corruption in normal use, or a documented hard-crash | 3 — **2 fixed (`5cd491b7`), 1 open (B1)** |
| **1 — High** | Real bug, or a contract that must be frozen/fixed before 1.0 | ~20 |
| **2 — Medium** | Worth doing around 1.0 | ~22 |
| **3 — Low** | Cleanup; can trail 1.0 | ~35 |

<sub>*Approximate; some items are consolidated across the per-unit and cross-cutting passes.</sub>

## Tier 0 — Blockers (must fix before 1.0)

### B1. SDIO RX buffer OOM → hard crash + reboot (documented, recurring, unmitigated)
**`sdkconfig:2728-2729` · `components/play_scheduler/play_scheduler_refresh.c:50` · `sdkconfig:1550-1551`**

`docs/sdio-rx-oom-crash.md` records **six hard crashes across three firmware builds and two IDF versions** (2026-05-04 → 2026-05-18), an `assert(*buf)` in the esp_hosted SDIO driver when the DMA-capable internal heap can't satisfy a streaming-mode RX allocation under concurrent refresh-paging + downloads + MQTT. The doc's chosen fix (option C) was **never applied**: `CONFIG_ESP_HOSTED_SDIO_OPTIMIZATION_RX_STREAMING_MODE` is still selected (the panic path), `SPIRAM_TRY_ALLOCATE_WIFI_LWIP` is still unset, `SPIRAM_MALLOC_RESERVE_INTERNAL` is still 32 KB, and `REFRESH_MAX_CONCURRENT` is still 2. Only the *diagnostic* heap snapshot landed — it logs the OOM, it does not prevent the reboot. **A device under normal multi-channel use can reboot unboundedly.** This is the single clearest thing standing between 0.x and a credible 1.0.

**Fix:** apply option C from the doc — flip to `CONFIG_ESP_HOSTED_SDIO_OPTIMIZATION_RX_MAX_SIZE=y`, enable `SPIRAM_TRY_ALLOCATE_WIFI_LWIP=y`, bump `SPIRAM_MALLOC_RESERVE_INTERNAL` to 65536, drop `REFRESH_MAX_CONCURRENT` to 1 — then build, flash, and soak under a full refresh cycle. (Requires hardware testing; this is the gating action.) The PSRAM-migration workstream in `docs/psram-migration-opportunities.md` is the complementary "raise the ceiling" follow-up.

### B2. ✅ FIXED — Heap overflow when Giphy returns more than `GIPHY_PAGE_LIMIT` entries
**`components/giphy/giphy_api.c:532-538` (caller buffer `components/giphy/giphy_refresh.c:344-345`)**

> **Fixed in `5cd491b7` (2026-06-02):** the parse loop is now bounded `… && parsed < GIPHY_PAGE_LIMIT`, and logs a warning if the server exceeds the limit. A deep-dive sweep confirmed this was the codebase's *only* unbounded server-array-into-fixed-buffer writer — all 7 museum parsers, makapix, and the channel parsers already bound their write index.

`giphy_fetch_page` loops `for (i = 0; i < array_size; i++)` writing `out_entries[parsed]`, where `array_size = cJSON_GetArraySize(data)` is **fully controlled by the remote server**, while the caller sizes `page_entries` to exactly 50. A response with 51+ items writes past the 64-byte-per-entry buffer and corrupts the heap. An external service must not be able to corrupt device memory in a 1.0.

**Fix:** clamp the loop — `for (int i = 0; i < array_size && parsed < GIPHY_PAGE_LIMIT; i++)` — and ideally pass an explicit `out_capacity` instead of relying on the implicit contract.

### B3. ✅ FIXED — Stack overflow in the captive-portal DNS server (reachable by any client on the setup AP)
**`components/wifi_manager/wifi_captive_portal.c:563-628`**

> **Fixed in `5cd491b7` (2026-06-02):** queries too large to leave room for the 16-byte answer (`len > (int)sizeof(tx_buffer) - 16`) are now dropped before the append. This was the only `recvfrom` in the first-party tree.

`recvfrom` fills `rx_buffer[128]` (so `len` can be 128); the handler then `memcpy`s `len` bytes into `tx_buffer[128]` and appends a 16-byte answer record at `answer_offset = len`. Any DNS query whose wire size exceeds 112 bytes (a long QNAME — trivial to send) overruns the stack buffer of the `dns_server` task. Reachable by any unauthenticated client connected to the setup AP.

**Fix:** bound-check before appending — `if (len > (int)sizeof(tx_buffer) - 16) continue;` — and reject `len > sizeof(rx_buffer)` defensively.

## Tier 1 — High priority before 1.0

### Correctness & robustness bugs

**H1. JPEG decoder: unbounded dimensions overflow 32-bit buffer math (HW path heap overrun)**
`components/animation_decoder/jpeg_animation_decoder.c:63-86` — `info.width/height` come from the untrusted SOF marker (museum/Giphy inputs) and are only checked for `== 0`; `rgb_buffer_size = (size_t)aligned_w * aligned_h * 3` wraps on RISC-V32, and IDF's own `*out_size` check wraps identically so it doesn't save you. A crafted JPEG can route a full-resolution decode into an undersized buffer. **Fix:** reject `width/height > MAX_JPEG_DIM` and compute the byte count in 64-bit on both the HW and SW paths.

**H2. `channel_cache_save` reads cache arrays without `cache->mutex` (races with merge)**
`components/channel_manager/channel_cache.c:551-668` vs `channel_cache_merge.c:300-303` — the debounced flush (holds only `registry_mutex`) can run while `merge_posts` does `free(cache->entries); cache->entries = …` under `cache->mutex`, yielding a torn read / use-after-free that writes a corrupt `.cache` or crashes. (Verifier narrowed this to the flush path; the `unregister` path is serialized.) **Fix:** snapshot counts + arrays under `cache->mutex`, then do SD I/O outside the lock.

**H3. Eviction NULL-derefs `evicted_ids` when its allocation fails**
`components/channel_manager/channel_cache_evict.c:39-41,99-105` — the loop increments `evicted` unconditionally but the post-mutex cleanup `lai_remove_entry(cache, evicted_ids[i], …)` has no NULL guard, so a failed `psram_malloc` under refresh pressure plus ≥1 eviction dereferences NULL. **Fix:** guard the cleanup loop with `if (evicted_ids)`.

**H4. Use-after-free in MQTT request/response correlation**
`components/makapix/makapix_api.c:152-184,324-342` — `pending_find()` returns the node pointer *after* releasing `s_pending_mutex`; the response callback then touches `pending->response_str` / `xSemaphoreGive(pending->done_sem)` while a timed-out waiter can `vSemaphoreDelete` + `free(pending)`. Narrow window, but most likely under exactly the congested-link conditions that produce both timeouts and late responses. **Fix:** do the match, the `response_str` write, and the give all under the mutex.

**H5. Use-after-free / use-after-scope when touch init times out**
`main/app_touch.c:534-640` — on the I2C-lockup timeout path the code `vSemaphoreDelete(ctx.done_sem)` while `touch_init_task` may still be blocked in `p3a_board_touch_init`; when it unblocks it writes `ctx->result` and gives the deleted semaphore, and the `streak>=1` branch has already returned (destroying `ctx`'s stack frame). Reachable in the exact scenario the rescue logic targets. **Fix:** heap/static-allocate `ctx` + `done_sem` and let the task free them; only `vSemaphoreDelete` on the success branch.

**H6. DNS callback writes to an out-of-scope stack variable after timeout**
`components/p3a_core/p3a_state_connectivity.c:264-337` — `dns_gethostbyname()` is handed `&dns_success` (a stack local); the function polls only 5 s but lwIP's DNS timeout is ~14 s, so a resolution landing in that window writes through a reclaimed stack frame. **Fix:** use `getaddrinfo()` (synchronous) or a heap/static context with a per-request token.

**H7. ✅ FIXED — `POST /rotation` did a single unbounded `httpd_req_recv` instead of `recv_body_json`**
`components/http_api/http_api_rest_settings.c:319-333` — unlike every sibling handler, it `psram_malloc(content_len+1)` with no `MAX_JSON` cap and read once (no drain loop), so a multi-segment body desyncs the keep-alive connection and a large `Content-Length` forces an unbounded allocation. **Fixed in `5cd491b7`** (surfaced by the B3 sibling sweep) — now uses `recv_body_json()` (MAX_JSON cap + chunked drain), matching its sibling handler.

**H8. Firmware OTA install leaks UI/OTA state on early `esp_https_ota` failures**
`components/ota_manager/ota_manager_install.c:222-237` — the `begin` / `get_img_desc` failure returns omit the `ota_exit_ui_mode()` + `p3a_state_exit_to_playback()` that every later error path performs, so a routine flaky-CDN failure wedges the OTA screen with playback stopped until reboot. **Fix:** mirror the later cleanup in both early returns.

**H9. Web-UI OTA marks an intact partition invalid before touching flash**
`components/ota_manager/ota_manager_webui.c:476-510` — `NVS_WEBUI_PARTITION_INVALID=1` is persisted (and LittleFS unmounted) *before* the partition lookup and size check; a too-large image or missing partition then returns with an intact partition flagged invalid, forcing an unnecessary recovery re-download on next boot. **Fix:** move the invalid-flag write to immediately before `esp_partition_erase_range`.

**H10. `app_main` panics on recoverable init via `ESP_ERROR_CHECK`**
`main/p3a_main.c:616,619,659,673` — `makapix_init`, `app_lcd_init`, `app_usb_init`, `connectivity_service_init` abort/reboot on any failure, while the surrounding inits log-and-continue. `app_lcd_init` forwards a transient `animation_player_start()` task-create failure, which would hard-loop the device instead of showing the error screen the no-SD path already proves is intended. **Fix:** replace these with log-and-degrade (or the fatal-error UI), reserving abort for truly unrecoverable invariants.

**H11. Playset editor `showToast` is an empty stub — all user feedback is silently dropped**
`webui/playset-editor.html:1916` — the page's only definition is `function showToast() {}`, so ~40 call sites (save/delete confirmations, every validation and network error) are no-ops. The `.toast-*` CSS exists, confirming this is a regression, not intentional. The editor ships with no operational feedback. **Fix:** delete the stub and add the real implementation (string `'error'`/`'success'` convention).

**H12. Stale WASM framebuffer view breaks PICO-8 rendering on heap growth**
`webui/pico8/pico8.js:388,505-591` — `fbView` is captured once but the fake08 build can grow the WASM heap (post-capture `_malloc`s), detaching the backing `ArrayBuffer`; the palette path is re-validated against `Module.HEAPU8.buffer` but the framebuffer path is not, so frames silently go black or `packet.set` throws. **Fix:** re-derive `fbView` (or guard `fbView.buffer !== Module.HEAPU8.buffer`) each `animate()` iteration.

**H13. Unsafe `vTaskDelete` of a status-publish task that may be mid-publish**
`components/makapix/makapix_connection.c:182-185` — the disconnect callback (invoked cross-task from the WiFi event task) force-deletes `s_status_publish_task` while it can be blocked inside `esp_mqtt_client_publish()`, risking a deadlock/leak of the MQTT client's internal mutex/heap. **Fix:** have the task self-exit on a flag/notification, or just stop the timer (it already no-ops when disconnected).

**H14. Dead — and dangerous — RGBA `animation_decoder_decode_next` path**
`components/animation_decoder/*` (+ `include/animation_decoder.h:85`) — the production pipeline only uses `…_decode_next_rgb`; the four RGBA `…_decode_next` impls have zero callers but write `W*H*4` into buffers the loader only ever sizes to `W*H*3`. Freezing this 4-bpp public API at 1.0 commits to a buffer-overrun footgun whose contract contradicts the pipeline. **Fix:** delete the RGBA entry point and its four impls before the API freezes.

### API & protocol stability — freeze decisions (these get *harder* after 1.0)

**H15. ✅ FIXED — API-version negotiation was broken end-to-end**
Resolved across `cf11bdb4`/`fcd7f9fb` (firmware side) and the `compat.js` single-sourcing change. `P3A_API_VERSION` is now injected globally in the root `CMakeLists.txt` (`add_compile_definitions` before `project()`), so `http_api`'s `/status` reports the real version (4) instead of the dead `#else api_version=1` fallback — which is now a hard `#error` on a missing macro rather than a silent 1. `webui/static/compat.js` is generated from `compat.js.in` via `configure_file()`, so its `REQUIRED_API` is single-sourced from `P3A_API_VERSION` and can no longer drift (it had sat at 1 while the API reached 4). The two stale-but-agreeing 1s that hid the check are gone, so the "Web UI outdated"/"Firmware outdated" banner now fires only on a genuine mismatch. The `P3A_API_VERSION` bump policy is documented at its definition in the root `CMakeLists.txt`.

**H16. REST URL surface is split across three naming eras with no version prefix**
`components/http_api/http_api.c:478-684` — newer endpoints live under `/api/*`, but a large legacy set sits at the root (`/config`, `/rotation`, `/channel`, `/channels/stats`, `/settings/*`, `/playsets`, `/status`, every `/action/*`, every `/ota/*`), there's a `@deprecated GET /channel` alongside `GET /playsets/active`, and `/api/museum/rate-limits` breaks the `{ok,data}` envelope every other endpoint uses. After 1.0 these become a frozen contract that third parties and the bundled UI both depend on. **Fix:** decide the convention now — either move JSON endpoints under a versioned `/api/v1/*` prefix (keeping thin redirects for one release) or explicitly document the current routes as the frozen 1.0 surface — and align the museum endpoint to the envelope. This is the most painful single thing to leave unfrozen.

**H17. Named-playset store *deletes* user files on version mismatch**
`components/play_scheduler/playset_store.c:212-218,354-358` — `playset_store_load`/`_list` `unlink` any `.playset` whose `version != PLAYSET_VERSION` (11). A firmware downgrade, or any future `PLAYSET_VERSION` bump, **silently destroys all saved playsets** with no recovery — the opposite of the forward-compat policy `pin_lists_manifest.c:95-99` already demonstrates (newer = read-only, never delete). **Fix:** make mismatch non-destructive (best-effort in-place upgrade via the existing `reserved[]` bytes for `<`, treat `>` as read-only); delete only on CRC/magic corruption.

**H18. `pin_lists.h` public contract describes removed v1 behavior**
`components/pin_lists/include/pin_lists.h:158-170,291-293,324-325` — the header says callers leave `post_id` at 0 and `pin_list_pin` "assigns the next monotonic id," but the v2 code requires the caller-supplied original post_id and the helpers reject `original_post_id <= 0` with `ESP_ERR_INVALID_ARG`. Following the documented contract fails at runtime. Freezing this header would lock in a lie. **Fix:** rewrite the docs to match v2; sweep out "monotonic id"/"next_post_id"/"pass 0 to synthesize" language.

**H19. Stand up automated tests + build CI**
Cross-cutting — there is **no host-side unit test, no `pytest_embedded`, no Unity `test/` app**, and `.github/workflows/` only publishes firmware to Pages (no `idf.py build` on push). An ~80k-line codebase committing to stability with no regression gate means any of the many documented refresh/concurrency fixes can silently regress. **Fix:** at minimum a GitHub Actions job that builds on push; ideally host-tests for the pure-logic units (channel_cache serialize/deserialize round-trip, `sd_path` sharding, playset DSL parse, JSON request parsers).

## Tier 2 — Medium (do around 1.0)

**Correctness / robustness**

- **M1. `ps_peek_next_available` permanently evicts cache entries** despite its "non-advancing" contract — `play_scheduler_pick.c:937-979` removes stale LAi entries during a *peek* and the snapshot doesn't restore `available_count`/the array. Add a read-only flag through the picker.
- **M2. Display-name lookups read shared channel state without the scheduler mutex** — `play_scheduler.c:153-207`, called from many tasks while `execute_playset` rewrites the array; can return a garbled name. Wrap in `s_state.mutex`.
- **M3. `load_new_format` trusts on-disk `ci_offset`/`lai_offset` with no bounds check** — `channel_cache.c:194-195,258-259`; CRC covers bit-flips, not a crafted/truncated-yet-consistent file. Validate offsets+sizes against `file_size`.
- **M4. `parse_artwork_from_json` fabricates `artwork_modified_at` from `time(NULL)`** — `playlist_manager.c:449-452` discards the stored value, defeating artwork-update detection and making reloads non-idempotent. Parse with the existing `parse_iso8601_utc`, and add the field to the serializer.
- **M5. Config store: unlocked read-modify-write of the whole blob (lost-update race)** — `config_store.c` has *no* mutex anywhere; concurrent `POST /settings` + a setter silently lose a setting (blast radius = the entire config). Add one mutex around load+save.
- **M6. `/static/*` and `/museum/*` map the raw URI to a filesystem path with no `..` guard** — `http_api_pages.c:267-284`; reject `..`/encoded variants like the upload handler already does.
- **M7. Upload-success response interpolates the raw filename into JSON unescaped** — `http_api_upload.c:473-486`; `a".gif` passes validation and breaks the JSON. Build the response with cJSON.
- **M8. PICO-8 audio stop/start races on the task handle** — `pico8_audio.c:150-171` nulls `s_task` after a fixed 100 ms delay with no handshake; a rapid restart spawns a second feed task writing the codec concurrently. Use a real completion signal.
- **M9. Event bus invokes handlers while holding its dispatch mutex** — `event_bus.c:80-92`; a blocking handler stalls all emitters and any re-entrant subscribe self-deadlocks (size-32 queue then silently drops). Snapshot subscribers under the lock, invoke after release.
- **M10. Parallel upscaler can busy-spin forever if a worker notification is lost** — `display_upscaler.c:507-521` has no overall deadline. Clear notify bits before notifying and cap the wait.
- **M11. `show_url` request handoff races a not-yet-cancelled prior download** — `show_url.c:592-611`; a binary-semaphore + shared-globals design can silently drop a queued "show this URL". Use a length-1 queue or guard the slot under a mutex.
- **M12. `fresh_boot` NVS erase list is wrong** — `fresh_boot.c:147-162` omits the live `"ota"` namespace and erases a nonexistent `"p3a_boot"`, so "simulate fresh boot" leaves OTA state behind. Fix the list and the header doc in one pass.
- **M13. BYOK API keys interpolated unencoded into request URLs** — `ham.c:149-158`, `smithsonian.c:191-195`; a key with `&`/`#`/`%`/space silently breaks the request. Run through the existing `ai_url_encode()`.
- **M14. AIC POST-DSL injects `term_id` unencoded into the Elasticsearch body** — `artic.c:518-536`; `term_id` traces to the user-influenced playset `identifier`, so `"`/`}` can malform/inject. Validate integer-only before the partitioned path.
- **M15. Web-UI OTA shared state mutated without the mutex readers hold** — `ota_manager_webui.c:458-461,477,648-652` torn-reads against `webui_ota_get_status`. Take `s_webui_ota.mutex`.
- **M16. `pin_list_unpin` filters by collision-prone `post_id`** — `pin_lists.c:944-947`; a DJB2 collision drops both rows from order.bin while deleting one artwork, orphaning a file. Match the exact `(src, source_id)` instead.
- **M17. Chunked Giphy downloads can cache truncated artwork as final** — `giphy_download.c:182-198` (root: `http_fetch.c:418-437` only checks truncation when `content_length > 0`); set `min_size`/`require_exact_length` for parity with `show_url`.
- **M18. `schedulePollSoon` leaks poll intervals on rapid actions** — `webui/index.html:2701-2707`; overlapping deferrals orphan a 4 s poller (unbounded background traffic). Track and clear the deferred timer.
- **M19. Rijks adapter fires 193 unbounded concurrent fetches** (`rijksmuseum.js:109-134`) and **its cursor walk can loop unbounded** on empty-but-paginated pages (`:144-156`). Route through `mapWithConcurrency` and break on a zero-item page.

**API / protocol & storage freeze items**

- **M20. NVS namespaces are an unversioned, undocumented cross-component contract** — `appcfg`, `wifi_config`, `makapix`, `ota`, `p3a_state`, `p3a_boot`; the `"ota"` namespace is even opened by two components with the keys defined as macros in one and string literals in the other (`ota_manager_webui.c:31-33` vs `p3a_board_fs.c:80,128`). Add a one-page registry, freeze the names, give each an owner, and consider a `schema_ver` key. (Also covers the `makapix` topic grammar and OTA health-flag keys — `makapix_mqtt.c:440-449`, `ota_manager_webui.c:31-33`.)
- **M21. Two parallel playset on-disk formats with different version policies** — the named store (`playset_store.c`, v11, hand-rolled 144-byte entries, deletes on mismatch) vs the active snapshot (`active_playset_store.c`, v1, records `spec_size`, opaque-blob, discards). Document both in one place and align mismatch policy (see H17).
- **M22. "Atomic" config write provides no crash recovery** — `config_store.c:204-250`; `cfg_new` is never read back on boot, so the temp-key dance only doubles flash writes. Either make it real journaling or drop it and fix the header's "atomic" wording.
- **M23. OTA has no anti-rollback / min-version gate** — `components/ota_manager`; a mis-tagged or downgraded release can install an image with an incompatible cache/NVS layout (channel_cache discards on mismatch). Add a version-ordering check and consider `CONFIG_BOOTLOADER_APP_ROLLBACK_ENABLE` + a post-boot self-test.
- **M24. Task-WDT survives only because PANIC is off** — `sdkconfig:1638-1639`; the JPEG progress-monitor fix landed but the libwebp arm (WDT #1) and `anim_loader` pinning are still deferred and no task feeds the WDT. Decide the 1.0 posture explicitly (document it as informational, or close the libwebp arm and turn PANIC on) — don't flip PANIC on blind.
- **M25. `p3a_render_set_channel_message` passes an enum as a raw `int`** — `p3a_render.h`/`p3a_state_channel.c:52` call it with a bare literal `4`; reorder the enum and every literal call site silently mismaps. Take `p3a_channel_msg_type_t` directly.
- **M26. OPC command work runs synchronously on the MQTT event task** — `makapix_mqtt.c:261-291` → `handle_set_rotation` does framebuffer reconfiguration on the MQTT loop, stalling keepalive. Enqueue to a worker task. (Verifier note: `set_paused` is already async; `set_rotation` is the real one.)
- **M27. `s_render` cache + several `s_state` fields are write-only** — `p3a_render.c`, `p3a_state.c`; ~80 lines of phantom bookkeeping (real rendering lives in `ugfx_ui.c`'s own statics) including ISO-8601 parsing for a value nobody reads. Delete, or wire to a real getter, before the surface freezes.
- **M28. `EXAMPLE_*` / `CONFIG_LCD_PIXEL_FORMAT_*` redefinitions leak through the board HAL header** — `p3a_board.h:302-330` `#define`/`#undef`s `CONFIG_*` symbols (owned by Kconfig) that `pico8_render.c` keys its pixel layout off, creating an include-order-dependent dual source of truth. Migrate to `P3A_*` names. (Carries the existing "TODO: Remove these after full migration".)
- **M29. Per-page inline JS is duplicated across Web UI pages with drifting conventions** — `showToast` is reimplemented in index/settings/ota and stubbed in the editor, with the signature diverging (boolean vs string `type`); `escapeHtml`/`escHtml`/`escapeAttr` likewise. This drift is *what let H11 regress unnoticed*. Extract `webui/static/common.js`.

## Tier 3 — Low (cleanup; can trail 1.0)

These are real but minor — grouped for brevity. Most are one-line fixes.

**Dead code / unused surface**
- Dead network-diagnostics stubs `log_all_netifs`/`verify_server_listening`/`format_sockaddr` — `http_api.c:688-714`.
- `mqtt_backoff_ms` computed (with jitter) but never read — `p3a_state_connectivity.c`.
- Dead `webp_decoder_*` externs (`static_image_decoder_common.h:51-57`); unused `cached_item_t` in `content_cache.h:19-25`; unused fields in the WebP decoder union.
- Dead legacy `animation_player` API (`find_animations_directory`, `get_next_asset_index`, `animation_player_add_file`, …) — `animation_player_loader.c`/`animation_player.c`. (Verifier: `free_sd_file_list`/`directory_has_animation_files` are *still live* — don't delete those.)
- `DISPLAY_UPSCALE_SINGLE_WORKER` hardwired dead branch — `display_upscaler.c:15-17,488-498`.

## Cross-cutting themes (the "decide once" refactors)

These aren't single bugs — they're patterns worth a deliberate decision before the code calcifies at 1.0.

**1. The 7 museum adapters duplicate a ~90–120 line refresh skeleton.**
`art_institution/museums/*.c` each repeat verbatim: buffer alloc with fallback, the identical `cache_size` clamp (`0→1024`, `>4096→4096`), the Si-hash track/teardown, orphan eviction, partial-content logic, and the SNTP-gated metadata save — and it has *already drifted* (Wellcome vs vam guard conditions differ). The inner merge/evict helpers are shared, but the orchestration isn't. **The same skeleton is echoed in `giphy_refresh.c` and `makapix_channel_refresh.c`.** Extract one `art_institution_run_refresh(…, fetch_page_cb)` driver. This is the single biggest maintainability liability in the firmware.

**2. `channel_offset` is implemented seven different ways and silently ignored by two museums.**
Wellcome and SMK `(void)channel_offset` while the other five use three mutually-incompatible wrap strategies — a user who sets an offset on those two channels gets a silent no-op with no UI signal. Add a `supports_offset` capability bit to the dispatch table so the inconsistency is discoverable, and converge the wrap math.

**3. The museum Web UI modules duplicate `getJson` + 429-reporting + `mapWithConcurrency` across all seven adapters.** Only the `museum:` literal varies. Extract `webui/museum/http.js`. (BYOK museums wrap it in `getJsonWithKey` with extra auth — structurally, not byte-for-byte, duplicated.)

**4. `makapix_promoted_https.c` hand-rolls `esp_http_client`** (`:142-228`) instead of using the shared `http_fetch` — with *weaker* behavior (no retry, no 429 handling, a silent-truncation read loop). The other Makapix paths already migrated; this one was missed. Replace with `http_fetch_to_buffer`.

**5. On-disk/NVS formats overwhelmingly "discard on version mismatch" with no migration framework and no documented policy.** That's fine for caches, dangerous for user data (see H17). Before 1.0, write down which persisted contracts are "cache, safe to invalidate" vs "user data, must migrate," and which are frozen.
